# C10 — Loop Convergence Constraint

## 목적

PEV 루프(loop-runner.sh)가 정해진 최대 반복 횟수 내에 수렴하지 못할 경우,
태스크 분할을 강제하여 무한 루프와 복잡도 폭발을 방지합니다.

## 기준

| 항목 | 기준 |
|------|------|
| 최대 반복 횟수 | 3회 (기본값, HARNESS.md `c10.max_iterations`로 변경 가능) |
| 검사 대상 | `.harness/loop/current.yaml` |
| 상태 필드 | `status: failed` 또는 `iteration > max_iterations` |
| 활성 Phase | development, stabilization, production |

## 체크 항목

- ✓ `status`가 `failed`가 아닌가?
- ✓ `iteration`이 `max_iterations` 이하인가?
- ✓ 루프 상태 파일이 존재하는가? (없으면 PASS — 루프 미사용)

## 위반 시 개선 방법

### 1. 태스크 분할

EXEC_PLAN의 목표(goal)가 너무 넓은 경우 분할:

```
# 나쁜 예: 여러 기능을 한 루프에서 검증
loop-runner.sh --run "bash scripts/constraint-check.sh && bash tests/all.sh"

# 좋은 예: 단일 검증 명령으로 분리
loop-runner.sh --run "bash scripts/constraint-check.sh"
loop-runner.sh --run "bash tests/specific.test.sh"
```

### 2. 검증 명령 재검토

검증 명령 자체에 오류가 있을 수 있습니다:

```sh
# 실패 원인 수동 확인
bash scripts/loop-runner.sh --check "검증_명령"

# 루프 상태 초기화 후 재시도
bash scripts/loop-runner.sh --reset
```

### 3. max_iterations 조정

불가피한 경우 `--max` 옵션으로 상한을 높일 수 있습니다 (권장하지 않음):

```sh
bash scripts/loop-runner.sh --max 5 --run "검증_명령"
```

## 측정

```bash
# C10 린터 직접 실행
bash linters/loop-convergence.sh

# 현재 루프 상태 확인
bash scripts/loop-runner.sh --status

# 루프 초기화
bash scripts/loop-runner.sh --reset
```

## 예외 처리

없음 — C10은 루프가 실행된 모든 phase에서 적용됩니다.
루프를 사용하지 않은 경우(current.yaml 없음) 자동으로 PASS 처리됩니다.
