# C03 — Task Completeness Constraint

## 목적

모든 작업의 요구사항과 목표가 명확하게 정의되어 있는지 확인합니다.

## 검증 항목

### 필수 필드

```yaml
---
task_type: general|feature|bugfix|refactor
title: "작업 제목"
done_condition:
  - "[auto] test: ..."
  - "[human] ..."
---
```

### task_type 유효 값

| 값 | 의미 |
|---|------|
| general | 일반적인 작업 |
| feature | 새 기능 추가 |
| bugfix | 버그 수정 |
| refactor | 코드 개선 |

### done_condition 형식

```markdown
done_condition:
  - "[auto] test: C03 PASS"        # 자동 검증 가능
  - "[auto] test: 커버리지 > 85%"
  - "[human] 코드 리뷰 완료"       # 사람의 확인 필요
  - "[human] 설계 문서 업데이트"
```

## 활성 Phase

planning, development, stabilization, production (모든 phase)

## 체크 항목

- ✓ task.md 파일이 존재하는가?
- ✓ task_type이 유효한 값으로 설정되어 있는가?
- ✓ title이 비어있지 않은가?
- ✓ done_condition에 최소 1개 이상의 항목이 있는가?

## 측정

```bash
# C03 실행
bash linters/task-completeness.sh

# done_condition 확인
grep -A 5 "^done_condition:" .harness/session/task.md
```

## 모범 사례

### 좋은 예

```yaml
---
task_type: feature
title: "사용자 인증 기능 추가"
done_condition:
  - "[auto] test: 테스트 코드 작성 및 통과"
  - "[auto] test: C03 PASS"
  - "[human] 보안 검토 완료"
---

## 개요
OAuth 2.0을 통한 로그인 기능 추가

## 단계
1. 준비: 관련 문서 검토
2. 구현: 인증 미들웨어 작성
3. 테스트: 통합 테스트
4. 검증: 보안 검토
```

### 나쁜 예

```yaml
---
task_type: ""          # ❌ 비어있음
title: ""              # ❌ 비어있음
done_condition: []     # ❌ 조건 없음
---
```
